# 代码库智能助手：五人并行开发任务规划

## 1. 项目定位

本项目是课程项目第一阶段的 MVP，目标是实现一个“本地代码库智能问答助手”。

用户输入一个本地项目目录，系统扫描其中的代码文件和 README，建立向量索引。用户提出代码相关问题后，系统检索相关代码片段，调用大语言模型生成回答，并展示回答依据的文件路径和行号。

第一阶段暂不实现以下内容：

- GitHub/Gitee 登录和 OAuth
- IDE 插件
- 多用户权限管理
- 在线同步远程仓库
- Issue、Pull Request、Commit 的深度语义检索
- Agent 自动修改代码
- 复杂的代码语法树分析

如果时间充足，可以将 commit 记录作为普通 Markdown 文本加入索引，作为附加功能，不影响主流程。

## 2. 最终演示流程

```text
输入本地项目路径
        |
        v
扫描代码和 README 文件
        |
        v
切分文本并建立向量索引
        |
        v
用户输入问题
        |
        v
检索相关代码片段
        |
        v
大模型根据上下文生成回答
        |
        v
前端展示回答、文件路径和行号
```

最终演示至少应支持以下问题：

1. “这个项目的启动入口在哪里？”
2. “用户登录流程是怎样的？”
3. “哪个文件负责数据库连接？”
4. “这个函数的作用是什么？”
5. “项目使用了哪些主要技术？”

## 3. Git 分支协作方式

### 3.1 分支结构

```text
main                       稳定演示版本
  |
develop                    集成开发分支
  |-- feature/repository   模块一：代码库扫描
  |-- feature/parser       模块二：文本切分与文档处理
  |-- feature/indexer      模块三：向量索引
  |-- feature/retrieval    模块四：检索与问答服务
  |-- feature/frontend     模块五：前端界面
```

### 3.2 初始化仓库

仓库目前只有远程地址时，由项目负责人执行一次：

```bash
git clone <GitHub或Gitee仓库地址>
cd codebase-rag-assistant
git checkout -b develop
mkdir backend frontend docs tests
git add .
git commit -m "chore: initialize project structure"
git push -u origin develop
```

之后五名成员分别从最新的 `develop` 创建自己的功能分支：

```bash
git fetch origin
git checkout develop
git pull origin develop
git checkout -b feature/自己的模块名
```

### 负责人首次启动任务

负责人需要先完成一次公共初始化，其他成员再开始创建自己的功能分支。建议将 Gitee 作为日常协作仓库，将 GitHub 作为同步备份仓库。

远程仓库配置如下：

```text
gitee  https://gitee.com/QiuZaoOP/code-seek-rag.git
github https://github.com/QiuZaoP/CodeSeek-RAG.git
```

如果本地目录还没有有效的 Git 提交历史，负责人按以下顺序执行：

```bash
cd codebase-rag-assistant
git init
git remote add gitee https://gitee.com/QiuZaoOP/code-seek-rag.git
git remote add github https://github.com/QiuZaoP/CodeSeek-RAG.git
git branch -M main
git add 并行开发任务规划.md
git commit -m "docs: add parallel development plan"
git push -u gitee main
git push -u github main
git checkout -b develop
git push -u gitee develop
git push -u github develop
```

如果本地已经执行过 `git init` 或已经存在远程配置，不要重复添加同名 remote。可以先检查：

```bash
git status
git remote -v
git branch --all
```

负责人还需要在 `develop` 上提交项目公共骨架，建议只创建目录和配置文件，不提前实现其他成员的业务代码：

```text
backend/
backend/api/
backend/models/
backend/repository/
backend/parser/
backend/indexer/
backend/qa/
backend/config/
frontend/
tests/
docs/
data/
.gitignore
.env.example
README.md
requirements.txt
```

公共骨架提交并推送后，负责人需要把以下内容发给全组：

1. 项目 Git 地址和推荐使用的远程仓库。
2. `develop` 分支名称。
3. 本文档中的五个模块和对应负责人。
4. 后端启动命令、前端启动命令和 Python 版本要求。
5. 模块之间不能随意修改的接口字段。

然后由每位成员从最新的 `develop` 创建自己的功能分支：

```bash
git fetch gitee
git checkout develop
git pull gitee develop
git checkout -b feature/自己的模块名
git push -u gitee feature/自己的模块名
```

负责人不需要直接编写五个模块的全部代码，主要负责公共约定、分支管理、接口变更协调和最终联调。

### 负责人开发期间的工作

- 确保所有成员从最新的 `develop` 创建分支。
- 维护接口文档和环境变量模板。
- 收集各模块的启动方式、测试命令和依赖版本。
- 每天检查功能分支是否能独立运行，发现接口冲突及时协调。
- 按“扫描 -> 切分 -> 索引 -> 检索问答 -> 前端”的顺序合并 Pull Request/Merge Request。
- 每次合并后在 `develop` 上运行一次基础联调。
- 只有完整流程通过后，才将 `develop` 合并到 `main` 并同步两个远程仓库。

同步稳定分支时可以执行：

```bash
git checkout main
git pull gitee main
git push github main
```

### 3.3 日常提交和合并规则

- 每个人只修改自己负责的目录，避免多人修改同一个文件。
- 每完成一个可运行的小功能就提交一次，不要积累大量未提交改动。
- 提交信息使用统一格式，例如 `feat: scan local repository files`。
- 每天开发前执行 `git pull origin develop`，发现冲突及时处理。
- 功能完成后，将自己的分支推送到远程，并提交 Pull Request/Merge Request 到 `develop`。
- 只有通过自测和接口检查后，才允许合并。
- `main` 只接受已经完成联调和演示验证的 `develop`。

推荐合并顺序：

1. 先合并公共目录、项目配置和接口文档的初始提交。
2. 合并模块一、模块二，验证文件扫描和文本切分。
3. 合并模块三，验证索引构建。
4. 合并模块四，验证检索和大模型问答。
5. 合并模块五，完成前后端联调。
6. 全组在 `develop` 上完成演示测试后，再合并到 `main`。

## 4. 五人模块分工

### 模块一：代码库扫描与项目管理

分支：`feature/repository`

负责人：成员一

#### 目标

接收本地项目路径，读取项目中的有效文件，并过滤不应进入知识库的目录和文件。

#### 负责目录

```text
backend/repository/
backend/api/project_api.py
tests/test_repository.py
```

#### 主要功能

- 接收项目路径并检查路径是否存在。
- 递归扫描项目文件。
- 默认支持 `.py`、`.java`、`.js`、`.ts`、`.vue`、`.go`、`.cpp`、`.c`、`.md`、`.json`、`.yaml` 文件。
- 忽略 `.git`、`node_modules`、`dist`、`build`、`.venv`、`venv`、`target`、`__pycache__` 等目录。
- 忽略超大文件和二进制文件。
- 返回文件路径、文件类型、文件大小和文件内容。

#### 对外接口

```http
POST /api/projects/load
Content-Type: application/json

{
  "project_path": "D:/demo-project"
}
```

返回：

```json
{
  "project_id": "demo-project",
  "project_path": "D:/demo-project",
  "file_count": 18,
  "files": [
    {
      "file_path": "src/main.py",
      "file_type": ".py",
      "size": 2450
    }
  ]
}
```

#### 验收标准

- 能扫描一个示例项目并返回文件列表。
- 能正确过滤依赖目录和二进制文件。
- 路径不存在、无权限、文件编码异常时返回清晰错误。
- 至少编写 3 个单元测试。

#### 可独立开发方式

在没有其他模块的情况下，使用本地临时目录和固定测试文件完成测试。模块二只依赖其返回的文件对象，不直接依赖扫描实现。

---

### 模块二：文本切分与文档处理

分支：`feature/parser`

负责人：成员二

#### 目标

将代码文件和 README 转换为带有来源信息的文本块，为向量化和检索提供统一数据格式。

#### 负责目录

```text
backend/parser/
backend/models/chunk.py
tests/test_parser.py
```

#### 主要功能

- 按文件逐行读取代码内容。
- 默认每 50 行切分一个代码块，保留 10 行重叠内容。
- README 和 Markdown 文件按标题或固定行数切分。
- 保留项目 ID、文件相对路径、起始行号、结束行号和文本内容。
- 跳过空文件和过大的单个文件。
- 提供统一的 `Chunk` 数据结构。

#### 统一数据结构

```json
{
  "chunk_id": "demo-project-src-main-py-1",
  "project_id": "demo-project",
  "file_path": "src/main.py",
  "start_line": 1,
  "end_line": 50,
  "content": "from fastapi import FastAPI\\n..."
}
```

#### 对外接口

```python
chunks = parser.parse_files(project_id, files)
```

其中 `files` 是模块一返回的文件元数据和内容列表，返回值是 `list[Chunk]`。

#### 验收标准

- 一个 120 行文件能够正确切分成多个文本块。
- 起始行号和结束行号正确。
- 相邻文本块具有重叠行。
- 文件路径信息不会丢失。
- 至少编写 4 个单元测试。

#### 可独立开发方式

不调用扫描模块，直接在测试中构造文件对象。模块三和模块四只依赖 `Chunk` 格式，不依赖切分代码的内部实现。

---

### 模块三：向量索引与索引管理

分支：`feature/indexer`

负责人：成员三

#### 目标

接收文本块，生成向量并保存到本地向量数据库，支持按项目建立和重建索引。

#### 负责目录

```text
backend/indexer/
backend/config/settings.py
tests/test_indexer.py
```

#### 推荐技术

- 向量数据库：Chroma
- 嵌入模型：项目允许联网时使用 API；否则使用本地轻量模型
- 第一阶段只要求本地持久化，不要求部署独立数据库服务

#### 主要功能

- 接收 `list[Chunk]`。
- 生成每个文本块的 embedding。
- 将 embedding、文本内容和元数据写入向量数据库。
- 支持按 `project_id` 删除旧索引并重新构建。
- 返回索引状态、文本块数量和耗时。

#### 对外接口

```python
indexer.build(project_id, chunks)
indexer.delete(project_id)
indexer.status(project_id)
```

HTTP 接口：

```http
POST /api/index/build
Content-Type: application/json

{
  "project_id": "demo-project"
}
```

返回：

```json
{
  "project_id": "demo-project",
  "status": "completed",
  "chunk_count": 42
}
```

#### 验收标准

- 能将测试文本块写入本地向量数据库。
- 相同项目重复构建时不会产生不可控的重复数据。
- 能查询索引状态。
- 没有 embedding API 时，能够通过 Mock embedding 完成自动化测试。
- 至少编写 3 个单元测试。

#### 可独立开发方式

使用模块二提供的固定 JSON 测试样例作为输入。embedding 和向量数据库都封装在本模块内，避免其他成员安装和调用细节不一致。

### 模块四：检索、上下文组装与大模型问答

分支：`feature/retrieval`

负责人：成员四

#### 目标

接收用户问题，检索最相关的文本块，组织提示词并调用大语言模型生成带引用的回答。

#### 负责目录

```text
backend/qa/
backend/api/chat_api.py
tests/test_qa.py
```

#### 主要功能

- 将用户问题转换成 query embedding。
- 从模块三的向量索引中检索 Top-K 文本块，默认 `K=5`。
- 对检索结果去重，并限制总上下文长度。
- 生成包含文件路径和行号的上下文。
- 调用大语言模型回答问题。
- 当上下文不足时，明确返回“未在当前代码库中找到足够依据”。
- 返回回答、引用来源和检索结果。

#### 对外接口

```http
POST /api/chat
Content-Type: application/json

{
  "project_id": "demo-project",
  "question": "用户登录流程是怎样的？",
  "top_k": 5
}
```

返回：

```json
{
  "answer": "用户登录首先在 ...",
  "sources": [
    {
      "file_path": "src/auth/login.py",
      "start_line": 12,
      "end_line": 38,
      "content": "..."
    }
  ]
}
```

#### 提示词要求

```text
你是代码库智能助手。请只根据提供的代码片段回答问题。
如果上下文没有足够信息，请明确说明，不要编造答案。
回答中必须引用相关文件路径和行号。

用户问题：{question}

相关代码片段：{context}
```

#### 验收标准

- 给定固定测试文本块和问题时，能够返回 Top-K 结果。
- 能构造包含来源信息的提示词。
- 大模型调用失败时返回可读错误，不导致服务崩溃。
- 使用 Mock LLM 时能完成自动化测试。
- 至少准备 5 个演示问题，并能展示引用文件。

#### 可独立开发方式

模块四通过接口或 Mock 访问模块三，不直接读取向量数据库内部文件。真实模型和 API Key 只在联调阶段配置，开发阶段使用 Mock 返回固定答案。

### 模块五：前端界面与联调入口

分支：`feature/frontend`

负责人：成员五

#### 目标

提供一个简单、可演示的 Web 界面，让用户完成项目加载、索引构建和代码问答。

#### 负责目录

```text
frontend/
docs/demo.md
```

#### 页面功能

- 输入本地项目路径。
- 点击“加载项目”，显示文件数量。
- 点击“建立索引”，显示构建状态和文本块数量。
- 输入代码问题并发送。
- 展示大模型回答。
- 展示引用文件、起止行号和代码片段。
- 对加载中、索引中、问答中、失败等状态提供明确提示。

#### 推荐技术

- React + Vite 或 Vue + Vite，五人统一选择一种。
- 使用 `fetch` 或 Axios 调用后端接口。
- 第一阶段不做复杂登录、路由和视觉设计。

#### 依赖的后端接口

```http
POST /api/projects/load
POST /api/index/build
GET  /api/index/status?project_id=demo-project
POST /api/chat
```

#### 验收标准

- 可以完整走通“加载项目 -> 建立索引 -> 提问 -> 查看引用”的流程。
- 请求失败时不会出现空白页面。
- 回答和来源信息不会互相遮挡。
- 至少准备一个演示项目和一份操作说明。

#### 可独立开发方式

前端先使用 `frontend/mock/mockApi.js` 模拟四个后端接口。模块四完成后只需切换 API 基地址，不改页面结构。

## 5. 公共接口和目录约定

项目建议使用如下目录：

```text
codebase-rag-assistant/
├── backend/
│   ├── main.py
│   ├── api/
│   ├── models/
│   ├── repository/
│   ├── parser/
│   ├── indexer/
│   ├── qa/
│   └── config/
├── frontend/
├── tests/
├── data/
│   └── chroma/
├── docs/
├── .env.example
├── .gitignore
├── requirements.txt
└── README.md
```

各模块必须遵守以下约定：

- 文件路径全部使用项目根目录下的相对路径。
- 行号从 1 开始。
- 模块间传递 JSON 或明确的数据类，不传递模块私有对象。
- API 返回统一格式：成功返回业务数据，失败返回 `{"detail": "错误说明"}`。
- 不把 API Key 提交到 Git，`.env` 必须加入 `.gitignore`。
- 数据库文件、缓存和模型文件默认不提交到 Git。
- 每个模块至少提供一个 README 或模块说明，写明启动方法和测试方法。

## 6. 团队开发时间安排

### 第 1 天：统一约定

- 确定 Python、前端框架和模型服务。
- 初始化 `main`、`develop` 和五条功能分支。
- 创建公共数据结构、接口文档、`.gitignore` 和环境变量模板。
- 准备一个小型示例代码库。

### 第 2 至 3 天：五人并行开发

- 成员一完成文件扫描。
- 成员二完成文本切分。
- 成员三完成向量索引。
- 成员四完成检索和 Mock 问答。
- 成员五完成前端页面和 Mock 接口。

### 第 4 天：分阶段合并

- 合并模块一和模块二，打通扫描到切分。
- 合并模块三，打通切分到索引。
- 合并模块四，打通索引到问答。
- 合并模块五，完成页面联调。

### 第 5 天：测试和汇报材料

- 使用固定问题测试回答质量。
- 检查来源文件和行号是否正确。
- 记录一次完整演示流程。
- 完善 README、系统架构图、分工表和项目总结。
- 将稳定版本合并到 `main`。

## 8. 项目最低交付物

最终仓库至少包含：

- 可运行的 FastAPI 后端。
- 可运行的前端问答页面。
- 本地代码扫描功能。
- 文本切分和来源行号记录。
- 向量索引和 Top-K 检索。

## 9. 合并完成定义

一个模块只有同时满足以下条件，才算完成并允许合并：

1. 代码位于约定目录，没有修改其他成员的模块目录。
2. 能在没有真实 API Key 的情况下通过 Mock 或单元测试。
3. 对外接口与本文档一致，或已在 Pull Request 中同步修改接口说明。
4. 有至少一条成功用例和一条失败用例。
5. 提交信息清晰，代码可以被其他成员运行。

项目整体只有在 `develop` 上完成一次完整演示后，才合并到 `main`。
